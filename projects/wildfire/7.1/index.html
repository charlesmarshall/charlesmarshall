Wildfire CMS version 7.1

Meet the new, clean and lean version of Wildfire. Time and design have moved on, so it was about time Wildfire did as well.

A lighter interface (some of which I've used on there) with a lot of the layout issues we've had in v6.pretty sorted out.

With every upgrade, we always try to make it faster & smarter and we've done both for this version.

That's a long one...

On our larger projects we started to encounter a few issues with just how many extra "modules" they had. On some sites, especially those with multiple languages, the left hand navigation would be immense.

The extreme examples of these lead to users struggling to find things, taking many seconds of scrolling down to find the right button to press. Now we've introduced a sub-navigation element that collapses and much tighter spacing.

Number love

As a bit of a number fan, it was about time we gave some love to the analytics part of the dashboard. Vastly improved graphs (google charts), much bigger and cleaner than before, there is also a couple of new ones added in.

With the ever increasing mobile usage, we've added in a infographic on the dash to show the percentage split for your website.

The other new graph is, of course, browser usage; so now you can monitor the decline of IE without leaving the CMS

The need for speed

In recent times there has been a tendancy towards more complicated sites, as what used to be the cutting edge becomes commoditised, more things become standard features.

Speed becomes more and more important. One of the slowest parts of the CMS has always been the tree structure, so it was about time that got checked out.

Tree-tastic

Probably the biggest single speed fix in this upgrade was one of the simpliest, once thought about it the other way around. As part of the tree models save it makes a few integratity & validation checks, one of those was to prevent cyclic trees.

It turned out that how it had been doing this previously was over-zealous to say the least. It built the entire content tree from the database in order to check that it hadn't been made a child of one of it's own children; on sites with large databases this causes memory issues.

What it now does instead is check the new parents upward stack to make sure its not in there. Although this takes a few more DB calls, these are based on indexed columns, so are much more efficient than an entire table scan.

Kids

As the content part fo the CMS operates like a tree, the interface need to let you view a nodes children, if it has any. Originally this was done by making a secondary database call, so on wide trees you can imagine how many that would be.

Swapping that over to a simple sub-query in the original SQL solved that in one go.

Permission

One of the features in Wildfire is user permissions, so you can restrict users from doing certain things; that of course means these permissions have to be checked in many places.

We do have a cache mechanism for that setup, so permissions are only checked once per render. The permissions operate on a blacklist, so unless you have been blocked from something, your permissions are empty.

Unfortunately, there was a slight problem. The cache was not recording empty permissions, so looking them up in the database every time. That is now fixed and it holds a placeholder regardless.

Attack of the tabs

In a similar vain to the long navigation issues, the content editing pages were just getting too much; large sites ended up with 10 or more tabs accross the top, each with losely related form fields.

Now it's much easier. Normally, no more than 3 top level tabs split in to very specific items: content, design & media.

Although we still need all of those fields, not every page is an event, or location, so now only the common demoninators are shown, extras are split in to accordions underneath.

Seeing is believing

In previous versions we've always had a 'preview' button, but this is a bit clunky and forces new windows / tabs to be opened. With the rise of mobile / tablets its become important to cater for those kind of devices, so we've included a customisable way of checking these in our CMS.

The design tab on the editing page gives you an iframed version of the page you are editing, as you are editing it. This can also be configured to show repsonsive versions of the page.

Obviously, the responsive preview only works for browsers that support media queries

Help

A real sticking point with building and supporting large CMS based sites is remember how to use then! Sometimes you don't need to change things for months at a time (bus routes, quarterly offers etc) and people forget.

Wildfire now contains help system for its modules (not all are written yet); by default it gives you the basics, but it can be extended per application to accomodate all of bespoke items.

How?

The system itself works from a simple json config version of the page. When you trigger the help it fetches the json file, goes to the first 'stage' of the help and uses jQuery to clone over the specified elements into an overlay.

It moves the cloned items to the position of the original and add highlighting.
